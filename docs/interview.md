# Интервью: Projects Manager

**Дата проведения:** 2026-04-09

---

### Q1: Платформа и интерфейс

**Вопрос:** Это приложение будет веб-приложением, десктопным, консольным (CLI) или другой формат?

**Ответ:** На первом этапе — локальное приложение, работа в текущей папке (PROJECTS_MANAGER). В дальнейшем планируется перенос в веб-приложение. Архитектура должна учитывать будущую миграцию на веб.

---

### Q2: Тип интерфейса на первом этапе

**Вопрос:** CLI, локальный веб-интерфейс на localhost или GUI-окно?

**Ответ:** Локальный веб-интерфейс на localhost с GUI. Интерфейс — минималистичный, но практичный. Детали отображения функционала определяются в ходе интервью.

---

### Q3: Технологический стек

**Вопрос:** Какой технологический стек предпочтителен? Frontend, backend?

**Ответ:** Пользователь предложил Next.js (frontend) + Node.js или Python (backend), выбор оптимального варианта — на усмотрение разработчика.

**Рекомендация:** Использовать **Next.js** как единый full-stack фреймворк:
- **Frontend:** Next.js (React) + TypeScript + Tailwind CSS
- **Backend:** Next.js API Routes (встроенный серверный слой на Node.js)
- **Хранение данных:** Markdown-файлы (чтение/запись через Node.js `fs`)

**Обоснование:**
- Next.js уже включает серверную часть (API Routes) — отдельный backend-сервер не нужен, что упрощает архитектуру
- Один язык (TypeScript) на всём стеке — проще поддерживать
- Node.js нативно работает с файловой системой (markdown-файлы) — Python здесь не даёт преимуществ
- При миграции на веб достаточно задеплоить тот же Next.js проект — минимум изменений
- Tailwind CSS обеспечит минималистичный и практичный интерфейс без лишней сложности

### Q4: Структура данных и хранение

**Вопрос:** Как устроена структура проектов? Где располагаются папки, какие файлы нужны?

**Ответ:**
- Все проекты располагаются в папке `G:/PROJECTS/` (включая само приложение PROJECTS_MANAGER)
- Информация по каждому проекту хранится в подпапке `docs/` внутри папки проекта
- Обязательные markdown-файлы для каждого проекта:
  - `PRD.md` — описание продукта / требования
  - `PLAN.md` — общий план, крупные этапы
  - `CURRENT_STAGES.md` — текущий статус, детализированные подэтапы

**Структура на диске (пример):**
```
G:/PROJECTS/
├── flat24/
│   └── docs/
│       ├── PRD.md
│       ├── PLAN.md
│       └── CURRENT_STAGES.md
├── new_dictionary/
│   └── docs/
│       ├── PRD.md
│       ├── PLAN.md
│       └── CURRENT_STAGES.md
└── ...
```

**Примечание:** Приложение PROJECTS_MANAGER располагается в `G:/PROJECTS/PROJECTS_MANAGER/` и читает/пишет файлы в `G:/PROJECTS/`.

### Q5: Этапы проекта и система планирования

**Вопрос:** Этапы фиксированные для всех проектов или произвольные? Какая иерархия?

**Ответ:**
- Набор этапов — **произвольный** для каждого проекта (не фиксированный шаблон)
- Используется **двухуровневая система планирования:**
  - `PLAN.md` — крупные этапы, общая картина работ по проекту ("крупные мазки")
  - `CURRENT_STAGES.md` — детализация, более мелкие подэтапы в рамках плана

**Иерархия:**
```
PLAN.md (верхний уровень)
├── Этап 1: Проектирование
├── Этап 2: Разработка MVP
├── Этап 3: Тестирование
└── ...

CURRENT_STAGES.md (детализация)
├── Этап 2: Разработка MVP
│   ├── [x] Настроить проект
│   ├── [x] Сверстать главную страницу
│   ├── [ ] Реализовать API
│   └── [ ] Подключить базу данных
└── ...
```

### Q6: Статусы этапов и прогресс выполнения

**Вопрос:** Какие статусы нужны? Нужен ли процент выполнения?

**Ответ:**
- Расширенная система статусов для этапов/подэтапов:
  - **Запланирован** — этап определён, работа не начата
  - **В работе** — активная работа над этапом
  - **На паузе** — работа приостановлена
  - **Завершён** — этап выполнен
  - **Отменён** — этап отменён, не будет выполняться

- **Процент выполнения** — показывает прогресс **текущей активной стадии**, а не всего проекта целиком
  - Формула: `(завершённые подэтапы текущей стадии / общее количество подэтапов текущей стадии) × 100%`
  - На дашборде отображается: название проекта + текущая стадия + процент выполнения этой стадии
  - Отменённые подэтапы исключаются из расчёта

**Пример карточки на дашборде:**
```
flat24
Стадия: Разработка MVP — 50%
██████████░░░░░░░░░░ 
Статус: В работе
```

### Q7: Дашборд (главная страница)

**Вопрос:** Что отображается на главной странице?

**Ответ:**
- Список всех проектов в виде карточек
- **Сортировка по умолчанию:** по последней активности (последний проект, с которым работал — наверху)
- **Фильтрация и сортировка:** по статусу, по активности
- Каждая карточка проекта отображает:
  - Название проекта
  - Текущая стадия (из PLAN.md)
  - Процент выполнения текущей стадии (из CURRENT_STAGES.md)
  - Статус проекта (В работе / На паузе / и т.д.)

### Q8: Страница проекта (детальный вид)

**Вопрос:** Что видим при клике на карточку проекта?

**Ответ:**
- При клике на карточку проекта — переход на страницу проекта
- Принцип: основная информация — на одной странице, детали — по клику (кнопки/ссылки → модальные окна)

**Содержимое страницы проекта:**

1. **Краткое описание проекта** — что это, для чего делается (вверху страницы)
2. **План выполнения** — все стадии из PLAN.md с визуальной отметкой текущего этапа
3. **Детализация текущей стадии** — подэтапы из CURRENT_STAGES.md с их статусами
4. **Кнопка "PRD"** — открывает содержимое PRD.md во всплывающем (модальном) окне

**Принцип навигации:**
- Основной обзор — сразу виден на странице (описание + план + текущая стадия)
- Детальная информация — доступна через кнопки/ссылки → открывается в модальных окнах
- Подход "progressive disclosure": не перегружать страницу, но дать доступ к деталям в один клик

### Q9: Источник описания проекта и определение проекта

**Вопрос:** Откуда берётся краткое описание? Как определить, что папка — это проект?

**Ответ:**

**Краткое описание проекта:**
- Хранится в файле `PRD.md` в виде YAML frontmatter-блока в начале файла
- Приложение читает метаданные из frontmatter и отображает описание на странице проекта

**Пример формата PRD.md:**
```markdown
---
name: "Flat24"
description: "Сервис для поиска и аренды квартир с удобным интерфейсом и фильтрацией"
status: "В работе"
created: "2025-12-01"
---

# PRD: Flat24

...полное содержимое PRD...
```

**Определение проекта:**
- Папка `G:/PROJECTS/` содержит только папки с проектами (без случайных директорий)
- **Маркер проекта:** наличие подпапки `docs/` внутри директории
- Если в папке нет `docs/` — приложение игнорирует её и не показывает в списке проектов

### Q10: Редактирование данных

**Вопрос:** Приложение только читает файлы или нужно и редактирование?

**Ответ:**
- Редактирование через интерфейс — **нужно**, но как вспомогательная функция (низкий приоритет)
- **Основной способ работы с файлами:** через Claude Code (создание, редактирование markdown-файлов напрямую)
- **Интерфейс приложения** — в первую очередь витрина/дашборд для просмотра состояния проектов
- Редактирование через GUI — страховочный вариант, используется редко

**Приоритеты реализации:**
1. **MVP (первая очередь):** чтение и отображение данных из markdown-файлов
2. **Вторая очередь:** базовое редактирование (смена статусов, отметка подэтапов как завершённых)
3. **Третья очередь (по необходимости):** полноценное редактирование (добавление этапов, создание проектов через UI)

### Q11: Формат файлов PLAN.md и CURRENT_STAGES.md

**Вопрос:** Какая структура у PLAN.md и CURRENT_STAGES.md? Какие поля нужны?

**Ответ:**

**PLAN.md — формат:**
- Составляется на основании PRD
- Каждый пункт плана: название + краткое описание
- Выполненные этапы отмечаются галочкой (markdown checkbox)
- Этапы — относительно крупные блоки работ

**Пример PLAN.md:**
```markdown
# План работ: Flat24

- [x] **Проектирование** — Составление PRD, определение архитектуры и стека технологий
- [x] **Дизайн** — Разработка макетов интерфейса и UI-kit
- [ ] **Разработка MVP** — Реализация основного функционала: поиск, фильтрация, карточки объектов
- [ ] **Тестирование** — Ручное и автоматизированное тестирование основных сценариев
- [ ] **Деплой и запуск** — Настройка инфраструктуры, CI/CD, первый релиз
```

**CURRENT_STAGES.md — формат:**
- Детализация каждого пункта плана на более мелкие задачи
- Каждый подэтап: название + статус + краткое описание (опционально)
- Дополнительные поля:
  - **Дата начала / дата завершения** — для отслеживания хронологии
  - **Примечания** — короткий комментарий, если нужен контекст

**Пример CURRENT_STAGES.md:**
```markdown
# Текущие этапы: Flat24

## Разработка MVP
Статус: В работе

- [x] Инициализация проекта (Next.js, настройка окружения) — завершён 2026-03-15
- [x] Вёрстка главной страницы — завершён 2026-03-22
- [ ] Реализация API поиска объектов — **в работе**
- [ ] Интеграция с базой данных — запланирован
- [ ] Компонент карточки объекта — запланирован
- [ ] Фильтрация и сортировка — запланирован

## Тестирование
Статус: Запланирован

- [ ] Написание unit-тестов для API
- [ ] Тестирование UI-компонентов
- [ ] E2E тестирование основных сценариев
```

**Примечание:** поле "ответственный" не используется — все проекты ведёт один человек. Формат файлов может быть скорректирован в дальнейшем.

### Q12: Создание нового проекта через интерфейс

**Вопрос:** Нужна ли кнопка "Новый проект"?

**Ответ:**
- **Да**, необходима кнопка "Новый проект" в интерфейсе
- При создании автоматически генерируется стандартная структура:

```
G:/PROJECTS/{имя_проекта}/
└── docs/
    ├── PRD.md          (шаблон с frontmatter)
    ├── PLAN.md         (пустой шаблон)
    └── CURRENT_STAGES.md (пустой шаблон)
```

- Все проекты создаются по единому шаблону — это обеспечивает единообразие и упрощает работу как через интерфейс, так и через Claude Code

### Q13: Дополнительный функционал

**Вопрос:** Нужны ли поиск, теги/категории, архив?

**Ответ:** Все три функции реализуются на первом этапе (MVP):

1. **Поиск** — быстрый поиск по названию проекта и содержимому файлов. Строка поиска на дашборде.

2. **Теги/категории** — группировка проектов по типу (например: "веб", "мобильное", "pet-project" и т.д.)
   - Теги хранятся в frontmatter `PRD.md` (поле `tags`)
   - На дашборде — фильтрация по тегам

3. **Архив** — возможность архивировать завершённые проекты
   - Архивированные проекты не отображаются на основном дашборде по умолчанию
   - Доступны через отдельный фильтр/раздел "Архив"
   - Статус архивации хранится в frontmatter `PRD.md` (поле `archived: true`)

**Финальный frontmatter PRD.md:**
```yaml
---
name: "Flat24"
description: "Сервис для поиска и аренды квартир"
status: "В работе"
tags: ["веб", "pet-project"]
archived: false
created: "2025-12-01"
last_updated: "2026-04-09"
---
```

### Q14: Определение последней активности и инициализация Git

**Вопрос:** Как определять последнюю активность проекта для сортировки?

**Ответ:**

**Последняя активность:**
- Определяется через поле `last_updated` в frontmatter `PRD.md`
- Обновляется при **любом** изменении файлов внутри папки проекта — как программных файлов, так и файлов в `docs/`
- Используется для сортировки проектов на дашборде (самый свежий — наверху)

**Инициализация Git:**
- При создании нового проекта через интерфейс — предлагать пользователю опцию инициализации Git-репозитория (`git init`)
- Опция не обязательная — пользователь может отказаться
- Если пользователь соглашается — выполняется `git init` в корне папки проекта и создаётся начальный коммит с шаблонными файлами

### Q15: Дизайн и визуальный стиль

**Вопрос:** Цветовая схема? Стиль? Иконки?

**Ответ:**

**Цветовая схема:**
- Тёмная и светлая темы с переключением
- Тёмная тема — мягкая, не чисто чёрная (тёмно-серые тона, например `#1a1a2e`, `#16213e`)

**Стиль интерфейса:**
- Простой, чистый, интуитивно понятный
- Ориентир: главная страница Zoom — лаконичный дизайн, много воздуха, без визуального шума
- Без эмодзи — вместо них линейные (outline) иконки

**Рекомендации по icon pack:**

1. **Lucide Icons** (рекомендуется) — форк Feather Icons, 1500+ иконок, линейный стиль, отличная интеграция с React/Next.js (`lucide-react`), активно развивается, бесплатный
2. **Phosphor Icons** — 6000+ иконок, 6 весов (thin, light, regular, bold, fill, duotone), гибкая настройка толщины, бесплатный
3. **Tabler Icons** — 4500+ иконок, линейный стиль, хорошая React-поддержка, бесплатный

**Рекомендация: Lucide Icons** — лёгкий, популярный в экосистеме Next.js/Tailwind, стиль максимально близок к чистому минимализму Zoom. Tree-shakeable — в бандл попадают только используемые иконки.
